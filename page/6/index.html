本地镜像管理

docker images:列出本地镜像

OPTIONS说明：

  • -a :列出本地所有的镜像（含中间映像层，默认情况下，过滤掉中间映像层）；
  • –digests :显示镜像的摘要信息；
  • -f :显示满足条件的镜像；
  • –format :指定返回值的模板文件；
  • –no-trunc :显示完整的镜像信息；
  • -q :只显示镜像ID。

docker rmi:删除镜像(一个或多个)

OPTIONS说明：

  • -f :强制删除；
  • –no-prune :不移除该镜像的过程镜像，默认移除；

docker tag:标记本地镜像,将其归入某一仓库

docker build:使用Dockerfile创建镜像

OPTIONS说明(常用的几个)：

  • -m :设置内存最大值；
  • –file :指定要使用的Dockerfile路径(默认是当前路径下)；
  • –pull :尝试去更新镜像的新版本；
  • -q :安静模式，成功后只输出镜像ID；
  • –rm :默认为true,设置镜像成功后删除临时容器(Dockerfile的每一行指令都会创建一个临时的容器)；
  • -t: 镜像的名字及tag，通常name:tag或者name格式,例如:-t NAME:TAG；

[PATH | URL]说明:给出命令执行的上下文,上下文可以是构建执行所在的本地路径,也可以是远程URL

容器

容器生命周期命令

docker run:创建一个新的容器并执行一个命令

OPTIONS说明(常用的几个)：

  • -d: 后台运行容器，并返回容器ID；
  • -i: 以交互模式运行容器，通常与 -t 同时使用；
  • -t: 为容器重新分配一个伪输入终端，通常与 -i 同时使用；
  • –name=”nginx-lb”: 为容器指定一个名称；
  • –link=[]: 添加链接到另一个容器；
  • –expose=[]: 开放一个端口或一组端口；
  • -P : 将所有公开的端口发布到随机端口;
  • -p : 指定端口或IP进行映射,例如:-p 8080:80;
  • -v : 给容器挂载存储卷，挂载到容器的某个目录

    例如:-v /data:/data,将主机的目录/data映射到容器的/data

  • –net : 容器网络设置,--net="bridge"

    bridge 使用docker daemon指定的网桥
    host //容器使用主机的网络
    container:NAME_or_ID
    none 容器使用自己的网络（类似–net=bridge），但是不进行配置

  • -w(–workdir=””) : 指定容器的工作目录,-w /workspace

docker rm : 删除一个或少个容器

  • -l :移除容器间的网络连接，而非容器本身
  • -v :-v 删除与容器关联的卷

Webpack4.0

使用webpack4时,确保使用node.js的版本 >= 8.9.4

webpack4新特性:

  1. mode属性

webpack需要设置mode属性，可以是development或production。例如：

webpack –mode development

针对开发模式提供的特性:

  • 浏览器调试
  • 注释、开发阶段的详细错误日志和提示
  • 快速和优化的增量构建机制、

针对生产模式提供的特性：

  • 开启所有的优化代码
  • 更小的bundle大小
  • 去掉只在开发阶段运行的代码
  • Scope hoisting和Tree-shaking
  1. 插件和优化

  1. 开箱即用的WebAssembly

WebAssembly（wasm）会带来运行时性能的大幅度提升，由于在社区的热毒，webpack4对它做了开箱即用的支持。你可以直接对本地的wasm模块进行import或者export操作，也可以通过编写loader来直接import C++、C或者Rust

  1. 支持多种模块类型

共5中：

  • JavaScript/auto：在webpack3里，默认开启对所有模块系统的支持，包括CommonJS、AMD、ESM。
  • JavaScript/esm:只支持ESM这种静态模块
  • JavaScript/dynamic:只支持CommonJS和AMD这种动态模块
  • json:只支持JSON数据,可以通过require和import来使用
  • WebAssembly/experimental:只支持wasm模块,目前处于实验阶段
  1. 0CJS

0CJS的含义是0配置,webpack4受到Parcel打包工具启发,尽可能的让开发者运行项目的成本变低。为了做到0配置，webpack4不在强制需要webpack.config.js作为打包的入口配置文件了，他默认的入口‘./src/’和默认出口‘./dist’，对小项目而言是福音（配置文件不推荐舍弃掉）。

  1. 新的插件系统

webpack4对插件系统进行了不少修改，提供了针对插件和钩子的新API。

变化如下：

  • 所有的hook有hooks对象统一管理，它将所有的hook作为可扩展的类属性。
  • 当添加插件时，必须提供一个插件名称。
  • 开发插件时，可以选择sync/callback/promise作为插件类型
  • 可以通过this.hooks = { myHooks:new SyncHook(…) }来注册hook了

服务端渲染

服务端渲染指的是将Vue组件渲染为服务端的HTML字符串，再返回给浏览器。服务端渲染的Vue应用程序也可以被认为是“同构”或“通用”。

SPA项目为什么需要SSR

加载首屏的时候需要下载许多资源,这导致了首屏白屏时间过长，影响用户体验。此外，由于使用JS渲染页面，不利于浏览器抓取页面内容。

SSR的好处

  • 更快的内容到达时间，减少首页白屏时间
  • 更好的SEO

路由跳转多次的问题

问题代码

原因:i标签的点击事件产生了冒泡,触发了router-link的点击事件,所以导致了路由跳转两次的问题

页面刷新vuex状态丢失

==问题原因:== 因为store里的数据是保存在运行内存中的,当页面刷新时，页面会重新加载vue实例，store里面的数据就会被重新赋值

解决思路

讲state里的数据保存一份到本地存储(localStorage,sessionStorage,cookie)中

==本地存储的选择:==

  1. localStorage是永久存储在本地(除非手动删除)
  2. sessionStorage是存储到当前页面关闭为止
  3. cookie则根据你设置的有效时间存储(不能存储大数据且不易读取)

解决方案一

由于state里的数据是响应式的,所以sessionStorage存储也要跟随变化。我们可以在mutation修改state的同时修改sessionStorage对应的存储

该方案存在一定的缺点,我们每次提交mutation的时候都要修改sessionStorage的值,但是我们只想在页面刷新前去存储,这样做会造成性能浪费

解决方案二

通过beforeupload事件在页面刷新时先触发,存储vuex的状态
